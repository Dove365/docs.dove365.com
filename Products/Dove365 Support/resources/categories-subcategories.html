Categories and Subcategories Guide

Keep support classification clean, useful, and reportable across all Cases.

Purpose

Consistent support classification

Categories and Subcategories help support users describe what a Case is about. Good classification improves triage, dashboards, trend analysis, and service improvement conversations.

Keep values short, clear, and support-focused. Too many categories makes reporting harder, not easier.
Design Principles

Build a practical taxonomy

01

Use categories for broad areas

Examples might include Access, Configuration, Data, Reporting, Integration, Billing, or General Enquiry.

02

Use subcategories selectively

Add subcategories only when they improve routing, analysis, or customer conversation.

03

Review values regularly

Retire unused values and merge duplicates before the list becomes noisy.

Administration

Maintain categories safely

✓

Create only when needed

Check for an existing category before creating a new one. Avoid spelling variants and synonyms.

✓

Deactivate instead of delete

When historical Cases use a value, deactivate it rather than deleting it so reporting remains understandable.

✓

Document meaning

For categories that are easy to confuse, maintain a short internal definition so support users classify consistently.

✓

Check reports after changes

Changing classification values affects trends, dashboards, and support analysis.

Governance

Avoid classification drift

  • Nominate owners for the category list.
  • Review new category requests before adding values.
  • Keep subcategories aligned to real support reporting needs.
  • Train users on when to choose each category.
  • Monitor “Other” or “General” usage because high volume usually means the taxonomy needs review.
